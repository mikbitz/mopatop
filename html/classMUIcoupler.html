Detailed Description

This class defines an interface for tow copes of the model running on different MPI threads.

On each MPI thread (currently just two allowed) define an interface for exchange of data between the threads
In each time step, the list of agents is checked to see whether data should be transferred between threads

check through all agents looking for exchanges and then transferring a subset of data as defined by push_data and pull_data

this gets annoyingly complex :(. When an agent leaves its original domain, it is assumed it will return at some point
i.e. each moving agent is a traveller. For this reason the original agent on the original domain is left untouched, but labelled as inactive
Any agents (either local ones, or travellers) that are set to leave their current domain in this timestep are initially labelled as leavers.
In the first part of the routine, all agents that are inactive have their ID stored. This is so that incoming data can be assigned to
returning agents when they come back from travel. Then local leaver agents have their data pushed out to the remote domain, labelled
with a zero to show that they are outgoing, and with their local agent ID (note this may coincide with a remote ID, but such remote agents
will not be "travellers"). Following this travellers on the current domain that are labelled as leaving (i.e. they are about to go home)
have their data pushed, now with a label larger than 1 to show that they are not local to this pushing thread (larger than one is used since the
labels are currently doubles, so guarantees that we can distinguish this value reliably from 0). These leaving agents are immediately deactivated
on the local thread (since they are about to go away to the remote MPI domain), and also re-labelled as non-leavers (since they cannot leave their
domain a second time, unless and until they return to the current domain).
All changes are now committed - i.e. they are sent to the remote domain - this implies a wait for the other thread to be ready.
In the second part of the routine, data can now be fetched - this is all done in one go, and the data stored ready for use: it needs to be allocated
to the right agents. First we deal with incoming travellers. On the remote thread, these were labelled with zero (since there they were local leavers)
so we pick out those arrivals first. We need to create a new agent on the local thread to store their data - this agent is added to the travellers present here
However, to save memory allocations, we can re-use any local inactive travellers - these were agents that were sent here, but have since gone home.
So first we set up a list of inactive local travellers and re-use these - if we run out of these, then we create some new agents. Each traveller is
added to the local vector of travellers (kept separate from the local list of agents) and assigned the ID from the remote domain. Then their schedules
are set up using a call to outwardTravel in agent.cpp . They are also labelled as needing to leave the domain at the end of their schedule.
Now we can add the returning travellers - we use their (local) ID and index, as transmitted with them from the remote domain to find their corresponding
local copy, activate it and copy in the data sent from the remote domain. The schedule for returning home is set with a call to inwardTravel .

mui::uniface<mui::mui_config>* MUIcoupler::flagInterface
private

a second interface, used to check whether there is any significant data to exchange

It seems if you do a commit on any interface at any particular time, any further commits are ignored
so this second interface is needed to exchange a flag between domains, so that they can check whether the other domain
has information to send. This avoids doing expensive loops over all agents if there is no data to send on either side
(since domain a cannot know if domain b has data to send until some data has been sent (!) and vice versa)
