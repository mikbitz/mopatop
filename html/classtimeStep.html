Detailed Description

A static class to set up the real-world times that apply to a timestep.

The idea here is that the code will use a timestep in seconds, but the user need not know this.
They can set values using a chosen time unit, and get the right number of timesteps in those using this class
The value of any time can be set in the relevant units by using the static variables e.g. to get a time of 8 hours

*double time_needed=8*timeStep::hour()

Similarly, to add the current value of the timestep to a given time, use timestep::deltaT()

double time= 4* timeStep::hour() + timeStep::deltaT()

Configure this once at the start of the code by creating an object and feeding it the parameterSettings object

parameterSettings parameters;
parameters.readParameters("../defaultParameterFile");
timeStep t(parameters);

A call to timeStep::update() is expected at the end of every model timestep. The model stepNumber can be used to calculate the number of hours and minutes, and the weekday since the start of the model run.
Each timestep the model updates the stepnumber held here for this purpose, so that the static stepNumber variable always holds the value of
the model step number, and this can be accessed from anywhere in the code. This is done by calling the timestep::update() method each timestep.
This adds one to the stepNumber, and then calculates the date given the starting date for the model (defaults to Mon 1 jan 1900, as a known day).
Days and months, minutes and hours are held as integers starting at zero to make date calculations easier.
At present there are no time zones available, so dates are calculated in nominal UTC (ignoring leap seconds, but allowing for leap years)
Similarly dates before 1752 are not correct as these predate the switch to the gregorian calendar.
The default initial date is Mon. 1 Jan. 1900.
NB Although one could use the ctime library, this proved to be unreliable in converting back and forth between dates in a tm structure and
seconds since 1970 that are held in a time_t, both in terms of results not always being consistent and in terms of calls to gmtime modifying
tm pointers not involved in the call. :(
An alternative would be to use the boost posix time and gregorian libraries, which work well, but boost can be a problem to compile against
as libraries can vary between systems. Maybe fixed by C++ 20 chrono?
