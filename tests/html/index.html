Introduction

This directory contains the unit tests for the mopatop model

Main ideas

The system uses the cppunit test system, version 1.14 to test each part of the model against some standard cases
For each class in the model, a header file is defined here containing tests of that class. Each header containes
a test suite, each of which runs through some of the methods defined in the classes to check that behave as expected
in a well defined and limited number of cases where the output can be known ahead of time. This helps to prevent some
of the more easily caught programming or conceptual errors. Each test suite is added to a testRunner defined in testing.cpp
Note that because some of the classes can depend on each other, running some of the tests might potentially affect others
(e.g. the timeStep class is static, so if its parameters are altered, subsequent tests that depend on the timestep might
be altered). This should be borne in mind when writing new tests.
When modifications are made to any of the classes, tests should be re-run to chekc that nothing has been broken. Any new
code not subject to testing should have tests added here as necessary. The tests themsevles have their own documentation
generated by Doxygen and kept in the html sub-directory.

Compiling

On a linux system with g++ installed just do
g++ -otests -fopenmp -std=c++17 -lcppunit ../timestep.cpp ../places.cpp ../disease.cpp ../agent.cpp testing.cpp
Note the current version requires g++>= and c++17 in order for the filesystem function to work (for creating new directories etc.)

Running

At present this is a simple command-line application - just type the executable name (tests above) and then return.
This will use the parameter file called testParameterFile in this directory.

./tests

The parameter file allows for a number of tests that depend on parameter settings.

Details

A testRunner is defined from the CppUnit::TextUi::TestRunner class. Each of the testsuites is added to this using

runner.addTest( testClass::suite() );

where testClass is defined in one of the header files. Each header file uses the CPPUNIT_TEST_SUITE macro to define a testsuite
which is a set of functions, each function containing one or more tests

CPPUNIT_TEST_SUITE( testClass );
//add tests defined below
CPPUNIT_TEST( function1 );
CPPUNIT_TEST( function2 );
CPPUNIT_TEST_SUITE_END();

If needed a setUp() and tearDown() function can be added, which will automatically create and destroy variables afresh at the start of
each test function, provided the class inherits from CppUnit::TestFixture e.g.

class placeTest : public CppUnit::TestFixture {
place* p;
public:
//persistent objects to use during testing
void setUp()
{
p = new place();
}
void tearDown()
{
delete p;
}
};

NB these function defintions must come BEFORE the above testsuite defintions.
In the test function typically asserts are used to check the result of of a test e.g.

void function1()
{
//default ID is zero
CPPUNIT_ASSERT(0==p->getID());
}

These are pass/fail tests, and will be reported giving the text of the failed test and line number in the code if needed
In addition tests may report text to stdout where required. Note the uses here of the pre-defined pointer p from the setUp(). Variables can also be defined as needed within each test function however.
